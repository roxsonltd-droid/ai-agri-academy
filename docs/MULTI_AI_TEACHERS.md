# MULTI AI TEACHERS — много агенти, оркестрация, личности, памет, специализации

Цел: **няколко AI „преподавателя“** с отделни **личности** и **специализации**, координирани чрез **LangGraph multi-agent** и **оркестратор**, с ясни слоеве **памет**. Текущият код има един минимален граф (`backend/agents/graph.py`); този документ задава **целевата архитектура** и връзката с **`agents/teachers_registry.py`**.

## Концепция

| Термин | Значение |
|--------|----------|
| **Teacher (преподавател)** | Един LLM агент с фиксиран **system prompt** + **роля** + **инструменти** (RAG slice, калкулатори, API). |
| **Orchestration** | Решение *кой* отговаря, *редът* на извикванията, *кога* да се прехвърли към друг учител или към човек. |
| **Multi-agent (LangGraph)** | Граф с **nodes** (учители, инструменти) и **edges** (условни или фиксирани), споделено **състояние**. |
| **Личност** | Име, тон, табута (напр. без емоджита), стил на обяснение — под **`TeacherPersona`** в регистъра. |
| **Специализация** | Тагове (`soil`, `crop_protection`, …) за **routing** и за **филтриране на RAG**. |
| **Памет** | Краткотрайна (сесия), дълготрайна (профил на студент), институционална (документи). |

## Роли в оркестрацията

### 1. Supervisor (маршрутизатор)

- **Вход:** съобщение на студента + контекст (сесия, избран курс).
- **Изход:** `next_teacher_id` или `clarify_question` или `human_handoff`.
- **Имплементация:** отделен node „`router`“ с малък модел или правила + LLM за неясни случаи.

### 2. Hub-and-spoke (център + специалисти)

- **Центърът** (генералист / „АгроМайнд“) събира отговори от специалисти или делегира един специалист.
- Подходящо, когато искате **един глас** към потребителя, но **вътрешно** няколко извиквания.

### 3. Паралелни специалисти → синтез

- Два учителя отговарят **паралелно** (напр. почва + защита на растенията), после node **`synthesize`** сглобява един отговор.

Изборът зависи от UX: **supervisor** е най-лесен за проследимост и логове.

## LangGraph — препоръчителна структура

```
          ┌──────────┐
          │  router  │
          └────┬─────┘
     ┌──────────┼──────────┐
     ▼          ▼          ▼
┌─────────┐ ┌─────────┐ ┌─────────┐
│ soil    │ │ crop    │ │ drones  │  … Teacher nodes
└────┬────┘ └────┬────┘ └────┬────┘
     └──────────┴──────────┘
                ▼
         ┌─────────────┐
         │  aggregate  │  (опционално)
         └──────┬──────┘
                ▼
              END
```

- **`StateGraph`**: споделено състояние виж **`agents/state.py`** (`MultiTeacherState`) — разширявайте с полета за LangChain съобщения, активен учител, бележки.
- **Checkpointer**: за **сесийна памет** и възобновяване — `SqliteSaver` (dev) / `PostgresSaver` (prod) в LangGraph 0.2+ (вижте актуалната документация на версията).
- **Tools per node:** всеки учител вижда само разрешени инструменти (напр. RAG с `filter_tags=specializations`).

## Отделни AI личности

- Регистър: **`agents/teachers_registry.py`** — за всеки `teacher_id`: име, кратко bio на български, **специализации**, **допълнение към system prompt** (не дублирайте целия глобален prompt — само диференциатор).
- **Глобални правила** (език, без емоджита, академичен тон) остават в **един** споделен блок; личността добавя само спецификата.
- **Frontend:** аватари, цветове, подпис „Проф. …“ — синхронизирайте с `teacher_id`.

## Специализации

- Таговете са **стабилни string slug-ове** (напр. `precision_ag`, `greenhouse`, `livestock`).
- Използват се за:
  - **routing** (supervisor prompt или embedding на intent → най-близък slug);
  - **RAG metadata filter** (документи/чанкове с `topics: ["soil", …]`);
  - **курсови пътеки** (модулът е свързан с тагове).

## Памет (memory) — слоеве

| Слой | Какво пази | Технология (пример) |
|------|------------|------------------------|
| **Сесийна** | Съобщения в чата, последен активен учител | LangGraph **checkpointer** (thread_id = session) |
| **Краткотрайна работна** | Scratchpad, междинни заключения между nodes | Полета в `MultiTeacherState` (`scratchpad`, `routing_notes`) |
| **Дълготрайна (студент)** | Цели, слаби теми, предпочитан език | Postgres таблица `user_ai_profile` + периодично обобщение от LLM |
| **Семантична (студент)** | „Какво вече знае студентът“ за персонализация | Embeddings в **Pinecone** / **pgvector** с namespace `user:{id}` |
| **Институционална** | Учебници, вътрешни PDF | Съществуващият **RAG** (`rag_facade`, knowledge API) |

**Принцип:** не вкарвайте целия чат в дълготрайната памет без **обобщение** — съхранявайте структурирани факти и източници.

## AI orchestration (практика)

1. **Intent** → router (текстова класификация или LLM с JSON schema).
2. **Delegate** → един или повече teacher nodes с ограничени tools.
3. **Validate** (опционално) → node за политики (PII, медицински/юридически рискове за агро съвети).
4. **Respond** → финален текст + опционално `citations` от RAG.

За **наблюдаемост**: логирайте `teacher_id`, `latency`, `token_usage` по node.

## Връзка с текущия код

| Компонент | Файл |
|-----------|------|

| Целево споделено състояние | `agents/state.py` — `MultiTeacherState` |
| HTTP за агенти | `api/agents_route.py` — `/api/v1/agents/run` |
| Чат без LangGraph | `core/ai_agent.py` — Проф. АгроМайнд + RAG |

Следваща имплементационна стъпка: нов граф `build_multi_teacher_graph()` в отделен модул, който импортира `TEACHERS` и checkpointer, без да чупи съществуващия `run_agro_agent`.

## Сигурност и качество

- **Handoff към човек** при ниска увереност или чувствителни теми (тор, пестициди — локални регулации).
- **Rate limit** по `user_id` + `teacher_id`.
- **Версионирайте** `system_prompt_extension_bg` в регистъра (git) като конфигурация.
